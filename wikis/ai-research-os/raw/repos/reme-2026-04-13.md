    47|
    48|? ReMe is a memory management framework designed for **AI agents**, providing
    49|both [file-based](#-file-based-memory-system-remelight) and [vector-based](#-vector-based-memory-system) memory systems.
    50|
    51|It tackles two core problems of agent memory: **limited context window** (early information is truncated or lost in long
    52|conversations) and **stateless sessions** (new sessions cannot inherit history and always start from scratch).
    53|
    54|ReMe gives agents **real memory** � old conversations are automatically compacted, important information is persistently
    55|stored, and relevant context is automatically recalled in future interactions.
    56|
    57|ReMe achieves state-of-the-art results on the LoCoMo and HaluMem benchmarks; see the [Experimental results](#experimental-results).
    58|
    59|<details>
    60|<summary><b>What you can do with ReMe</b></summary>
    61|
    62|<br>
    63|
    64|- **Personal assistant**: Provide long-term memory for agents like [QwenPaw](https://github.com/agentscope-ai/CoPaw),
    65|  remembering user preferences and conversation history.
    66|- **Coding assistant**: Record code style preferences and project context, maintaining a consistent development
    67|  experience across sessions.
    68|- **Customer service bot**: Track user issue history and preference settings for personalized service.
    69|- **Task automation**: Learn success/failure patterns from historical tasks to continuously optimize execution
    70|  strategies.
    71|- **Knowledge Q&A**: Build a searchable knowledge base with semantic search and exact matching support.
    72|- **Multi-turn dialogue**: Automatically compress long conversations while retaining key information within limited
    73|  context windows.
    74|
    75|</details>

    79|## ? File-based memory system (ReMeLight)
    80|
    81|> Memory as files, files as memory.
    82|
    83|Treat **memory as files** � readable, editable, and copyable.
    84|[QwenPaw](https://github.com/agentscope-ai/CoPaw) integrates long-term memory and context management by inheriting from
    85|`ReMeLight`.
    86|
    87|| Traditional memory system | File-based ReMe      |
    88||---------------------------|----------------------|
    89|| ?? Database storage      | ? Markdown files    |
    90|| ? Opaque                 | ? Always readable   |
    91|| ? Hard to modify          | ?? Directly editable |
    92|| ? Hard to migrate        | ? Copy to migrate   |
    93|
    94|```
    95|working_dir/
    96|??? MEMORY.md              # Long-term memory: persistent info such as user preferences
    97|??? memory/
    98|?   ??? YYYY-MM-DD.md      # Daily journal: automatically written after each conversation
    99|??? dialog/                # Raw conversation records: full dialog before compression
   100|?   ??? YYYY-MM-DD.jsonl   # Daily conversation messages in JSONL format
   101|??? tool_result/           # Cache for long tool outputs (auto-managed, expired entries auto-cleaned)
   102|    ??? <uuid>.txt
   103|```
   104|

   156|#### Python usage
   157|
   158|```python
   159|import asyncio
   160|
   161|from reme.reme_light import ReMeLight
   162|
   163|async def main():
   164|    # Initialize ReMeLight
   165|    reme = ReMeLight(
   166|        default_as_llm_config={"model_name": "qwen3.5-35b-a3b"},
   167|        # default_embedding_model_config={"model_name": "text-embedding-v4"},
   168|        default_file_store_config={"fts_enabled": True, "vector_enabled": False},
   169|        enable_load_env=True,
   170|    )
   171|    await reme.start()
   172|
   173|    messages = [...]  # List of conversation messages
   174|
   175|    # 1. Check context size (token counting, determine if compaction is needed)
   176|    messages_to_compact, messages_to_keep, is_valid = await reme.check_context(
   177|        messages=messages,
   178|        memory_compact_threshold=90000,  # Threshold to trigger compaction (tokens)
   179|        memory_compact_reserve=10000,  # Token count to reserve for recent messages
   180|    )
   181|
   182|    # 2. Compact conversation history into a structured summary
   183|    summary = await reme.compact_memory(
   184|        messages=messages,
   185|        previous_summary="",
   186|        max_input_length=128000,  # Model context window (tokens)
   187|        compact_ratio=0.7,  # Trigger compaction when exceeding max_input_length * 0.7
   188|        language="zh",  # Summary language (e.g., "zh" / "")
   189|    )
   190|
   191|    # 3. Compact long tool outputs (prevent tool results from blowing up context)
   192|    messages = await reme.compact_tool_result(messages)
   193|
   194|    # 4. Pre-reasoning hook (auto compact tool results + check context + generate summaries)
   195|    processed_messages, compressed_summary = await reme.pre_reasoning_hook(
   196|        messages=messages,
   197|        system_prompt="You are a helpful AI assistant.",
   198|        compressed_summary="",
   199|        max_input_length=128000,
   200|        compact_ratio=0.7,
   201|        memory_compact_reserve=10000,
   202|        enable_tool_result_compact=True,
   203|        tool_result_compact_keep_n=3,
   204|    )
   205|
   206|    # 5. Persist important memory to files (writes to memory/YYYY-MM-DD.md)
   207|    summary_result = await reme.summary_memory(
   208|        messages=messages,
   209|        language="zh",
   210|    )
   211|
   212|    # 6. Semantic memory search (vector + BM25 hybrid retrieval)
   213|    result = await reme.memory_search(query="Python version preference", max_results=5)
   214|
   215|    # 7. Create in-session memory instance (manages context for one conversation)
   216|    memory = reme.get_in_memory_memory()  # Auto-configures dialog_path
   217|    for msg in messages:
   218|        await memory.add(msg)
   219|    token_stats = await memory.estimate_tokens(max_input_length=128000)
   220|    print(f"Current context usage: {token_stats['context_usage_ratio']:.1f}%")
   221|    print(f"Message token count: {token_stats['messages_tokens']}")
   222|    print(f"Estimated total tokens: {token_stats['estimated_tokens']}")
   223|
   224|    # 8. Mark messages as compressed (auto-persists to dialog/YYYY-MM-DD.jsonl)
   225|    # await memory.mark_messages_compressed(messages_to_compact)
   226|
   227|    # Shutdown ReMeLight
   228|    await reme.close()
   229|
   230|if __name__ == "__main__":
   231|    asyncio.run(main())
   232|```

   238|### Architecture of the file-based ReMeLight memory system
   239|
   240|#### Context data structure
   241|
   242|```mermaid
   243|flowchart TD
   244|    A[Context] --> B[compact_summary]
   245|    B --> C[dialog path guide + Goal/Constraints/Progress/KeyDecisions/NextSteps]
   246|    A --> E[messages: full dialogue history]
   247|    A --> F[File System Cache]
   248|    F --> G[dialog/YYYY-MM-DD.jsonl]
   249|    F --> H[tool_result/uuid.txt N-day TTL]
   250|```
   251|
   252|---
   253|
   254|[MemoryManager](https://github.com/agentscope-ai/CoPaw/blob/main/src/copaw/agents/memory/reme_light_memory_manager.py)
   255|inherits `ReMeLight` and integrates its memory capabilities into the agent reasoning loop:
   256|
   257|```mermaid
   258|graph LR
   259|    Agent[Agent] -->|Before each reasoning step| Hook[pre_reasoning_hook]
   260|    Hook --> TC[compact_tool_result<br>Compact tool outputs]
   261|    TC --> CC[check_context<br>Token counting]
   262|    CC -->|Exceeds limit| CM[compact_memory<br>Generate summary]
   263|    CC -->|Exceeds limit| SM[summary_memory<br>Async persistence]
   264|    SM -->|ReAct + FileIO| Files[memory/*.md]
   265|    CC -->|Exceeds limit| MMC[mark_messages_compressed<br>Persist raw dialog]
   266|    MMC --> Dialog[dialog/*.jsonl]
   267|    Agent -->|Explicit call| Search[memory_search<br>Vector+BM25]
   268|    Agent -->|In - session| InMem[ReMeInMemoryMemory<br>Token-aware memory]
   269|    InMem -->|Compress/Clear| Dialog
   270|    Files -.->|FileWatcher| Store[(FileStore<br>Vector+FTS index)]
   271|    Search --> Store
   272|```

   274|---
   275|
   276|#### 1. `check_context` � context checking
   277|
   278|[ContextChecker](reme/memory/file_based/components/context_checker.py) uses token counting to determine whether the
   279|context exceeds thresholds and automatically splits messages into a "to compact" group and a "to keep" group.
   280|
   281|```mermaid
   282|graph LR
   283|    M[messages] --> H[AsMsgHandler<br>Token counting]
   284|    H --> C{total > threshold?}
   285|    C -->|No| K[Return all messages]
   286|    C -->|Yes| S[Keep from tail<br>reserve tokens]
   287|    S --> CP[messages_to_compact<br>Earlier messages]
   288|    S --> KP[messages_to_keep<br>Recent messages]
   289|    S --> V{is_valid<br>Tool calls aligned?}
   290|```
   291|
   292|- **Core logic**: keep `reserve` tokens from the tail; mark the rest as messages to compact.
   293|- **Integrity guarantee**: preserves complete user-assistant turns and tool_use/tool_result pairs without splitting
   294|  them.
   295|
   296|---
   297|
   298|#### 2. `compact_memory` � conversation compaction
   299|
   300|[Compactor](reme/memory/file_based/components/compactor.py) uses a ReActAgent to compact conversation history into a *
   301|*structured context summary**.
   302|
   303|```mermaid
   304|graph LR
   305|    M[messages] --> H[AsMsgHandler<br>format_msgs_to_str]
   306|    H --> A[ReActAgent<br>reme_compactor]
   307|    P[previous_summary] -->|Incremental update| A
   308|    A --> S[Structured summary<br>Goal/Progress/Decisions...]
   309|```
   310|
   311|**Summary structure** (context checkpoints):
   312|
   313|| Field                 | Description                                                                             |
   314||-----------------------|-----------------------------------------------------------------------------------------|
   315|| `## Goal`             | User goals                                                                              |
   316|| `## Constraints`      | Constraints and preferences                                                             |
   317|| `## Progress`         | Task progress                                                                           |
   318|| `## Key Decisions`    | Key decisions                                                                           |
   319|| `## Next Steps`       | Next step plans                                                                         |
   320|| `## Critical Context` | Critical data such as file paths, function names, error messages, etc.                  |
   321|
   322|- **Incremental updates**: when `previous_summary` is provided, new conversations are merged into the existing summary.
   323|- **Thinking enhancement**: with `add_thinking_block=True` (default), a reasoning step is added before generating the
   324|  summary to improve quality.
   325|
   326|---
   327|
   328|#### 3. `summary_memory` � persistent memory
   329|
   330|[Summarizer](reme/memory/file_based/components/summarizer.py) uses a **ReAct + file tools** pattern so that the AI can
   331|decide what to write and where to write it.
   332|
   333|```mermaid
   334|graph LR
   335|    M[messages] --> A[ReActAgent<br>reme_summarizer]
   336|    A -->|read| R[Read memory/YYYY-MM-DD.md]
   337|    R --> T{Reason: how to merge?}
   338|    T -->|write| W[Overwrite]
   339|    T -->|edit| E[Edit in place]
   340|    W --> F[memory/YYYY-MM-DD.md]
   341|    E --> F
   342|```
   343|
   344|**File tools** ([FileIO](reme/memory/file_based/tools/file_io.py)):
   345|
   346|| Tool    | Function              |
   347||---------|-----------------------|
   348|| `read`  | Read file content     |
   349|| `write` | Overwrite file        |
   350|| `edit`  | Find-and-replace edit |
   351|
   352|---
   353|
   354|#### 4. `compact_tool_result` � tool result compaction
   355|
   356|[ToolResultCompactor](reme/memory/file_based/components/tool_result_compactor.py) addresses the problem of long tool
   357|outputs bloating the context. It applies two different truncation strategies depending on whether a message falls within
   358|the `recent_n` window:
   359|
   360|```mermaid
   361|graph LR
   362|    M[messages] --> B{Within recent_n?}
   363|    B -->|Yes - recent| C[Low truncation recent_max_bytes=100KB<br>Save full content to tool_result/uuid.txt<br>Hint: 'Read from line N']
   364|    B -->|No - old| D[High truncation old_max_bytes=3KB<br>Reference existing file<br>More aggressive truncation]
   365|    C --> E[cleanup_expired_files<br>Delete expired files]
   366|    D --> E
   367|```
   368|
   369|| Parameter          | Default               | Description                                                                                                                   |
   370||--------------------|-----------------------|-------------------------------------------------------------------------------------------------------------------------------|
   371|| `recent_n`         | `1`                   | Minimum number of trailing consecutive tool-result messages treated as "recent" (use low truncation)                          |
   372|| `recent_max_bytes` | `100 * 1024` (100 KB) | Truncation threshold for recent messages; content beyond this is saved to `tool_result/` with a file path and start-line hint |
   373|| `old_max_bytes`    | `3000` (3 KB)         | Truncation threshold for older messages; truncation is more aggressive                                                        |
   374|| `retention_days`   | `3`                   | Number of days to retain tool result files; expired files are auto-cleaned                                                    |
   375|
   376|- **Auto cleanup**: expired files (older than `retention_days`) are deleted automatically during `start` / `close` /
   377|  `compact_tool_result`.
   378|
   379|---
   380|
   381|#### 5. `memory_search` � memory retrieval
   382|
   383|[MemorySearch](reme/memory/file_based/tools/memory_search.py) provides **vector + BM25 hybrid retrieval**.
   384|
   385|```mermaid
   386|graph LR
   387|    Q[query] --> E[Embedding<br>Vectorization]
   388|    E --> V[vector_search<br>Semantic similarity]
   389|    Q --> B[BM25<br>Keyword matching]
   390|    V -->|" weight: 0.7 "| M[Deduplicate + weighted merge]
   391|    B -->|" weight: 0.3 "| M
   392|    M --> F[min_score filter]
   393|    F --> R[Top-N results]
   394|```
   395|
   396|- **Fusion mechanism**: vector weight 0.7 + BM25 weight 0.3 � balancing semantic similarity and exact matches.
   397|
   398|---
   399|
   400|#### 6. `ReMeInMemoryMemory` � in-session memory
   401|
   402|[ReMeInMemoryMemory](reme/memory/file_based/reme_in_memory_memory.py) extends AgentScope's `InMemoryMemory` to provide
   403|token-aware memory management and raw conversation persistence.
   404|
   405|```mermaid
   406|graph LR
   407|    C[content] --> G[get_memory<br>exclude_mark=COMPRESSED]
   408|    G --> F[Filter out compressed messages]
   409|    F --> P{prepend_summary?}
   410|    P -->|Yes| S[Prepend previous summary]
   411|    S --> O[Output messages]
   412|    P -->|No| O
   413|    M[mark_messages_compressed] --> D[Persist to dialog/YYYY-MM-DD.jsonl]
   414|    D --> R[Remove from memory]
   415|```
   416|
   417|| Function                         | Description                                              |
   418||----------------------------------|----------------------------------------------------------|
   419|| `get_memory`                     | Filter messages by mark and auto-append summary          |
   420|| `estimate_tokens`                | Estimate token usage of the context                      |
   421|| `state_dict` / `load_state_dict` | Serialize/deserialize state (session persistence)        |
   422|| `mark_messages_compressed`       | Mark messages compressed and persist to dialog directory |
   423|| `clear_content`                  | Persist all messages before clearing memory              |
   424|
   425|**Raw conversation persistence**: When messages are compressed or cleared, they are automatically saved to
   426|`{dialog_path}/{date}.jsonl` with one JSON-formatted message per line.
   427|
   428|---
   429|
   430|#### 7. `pre_reasoning_hook` � pre-reasoning processing
   431|
   432|This is a unified entry point that wires all the above components together and automatically manages context before each
   433|reasoning step.
   434|
   435|```mermaid
   436|graph LR
   437|    M[messages] --> TC[compact_tool_result<br>Compact long tool outputs]
   438|    TC --> CC[check_context<br>Compute remaining space]
   439|    CC --> D{messages_to_compact<br>Non-empty?}
   440|    D -->|No| K[Return original messages + summary]
   441|    D -->|Yes| V{is_valid?}
   442|    V -->|No| K
   443|    V -->|Yes| CM[compact_memory<br>Sync summary generation]
   444|    V -->|Yes| SM[add_async_summary_task<br>Async persistence]
   445|    CM --> R[Return messages_to_keep + new summary]
   446|```
   447|
   448|**Execution flow**:
   449|
   450|1. `compact_tool_result` � compact long tool outputs for all messages except the most recent
   451|   `tool_result_compact_keep_n`.
   452|2. `check_context` � check whether the context exceeds limits (remaining space = threshold minus tokens used by system
   453|   prompt and compressed summary).
   454|3. `compact_memory` � generate compact summary (sync), appended into `compact_summary`.
   455|4. `summary_memory` � persist memory to `memory/*.md` (async in the background, non-blocking).
   456|
   457|| Key parameter                | Default | Description                                                                         |
   458||------------------------------|---------|-------------------------------------------------------------------------------------|
   459|| `tool_result_compact_keep_n` | `3`     | Skip tool result compaction for the most recent N messages (preserve full content)  |
   460|| `memory_compact_reserve`     | `10000` | Token count to reserve for recent messages; messages beyond this trigger compaction |
   461|| `compact_ratio`              | `0.7`   | Compaction threshold ratio: `max_input_length � compact_ratio � 0.95`               |
   462|

   465|## ?? Vector-based memory system
   466|
   467|[ReMe Vector Based](reme/reme.py) is the core class for the vector-based memory system. It manages three types of
   468|memories:
   469|
   470|| Memory type           | Use case                                                          |
   471||-----------------------|-------------------------------------------------------------------|
   472|| **Personal memory**   | Records user preferences and habits                               |
   473|| **Procedural memory** | Records task execution experience and patterns of success/failure |
   474|| **Tool memory**       | Records tool usage experience and parameter tuning                |
