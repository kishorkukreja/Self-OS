# GitHub Spec Kit — Spec-Driven Development Toolkit

## Summary

GitHub's **Spec Kit** is an open-source toolkit for **Spec-Driven Development (SDD)**: a workflow where specifications become first-class executable artifacts for coding agents rather than throwaway planning documents. The toolkit provides the `specify` CLI, project scaffolding, agent slash commands/skills, templates for specifications/plans/tasks, and integrations for multiple AI coding agents. Its positioning is directly aligned with moving from "vibe coding" toward predictable, governed agent implementation.

The core workflow is: establish project principles, write the product specification, produce a technical plan, break that plan into tasks, then ask an AI coding agent to implement against those artifacts. This makes the repo especially relevant to Self-OS/taskOS because it overlaps with the user's existing pattern of `/to-prd`, `/to-issues`, day/night shifts, and QA-gated agent implementation.

## Key points

- **Spec-first agent workflow:** Spec Kit treats requirements/specs as executable inputs for implementation rather than static docs.
- **Governance before implementation:** The `/speckit.constitution` step creates project principles and development guidelines before specs/tasks are generated.
- **Agent command lifecycle:** The documented lifecycle uses `/speckit.specify`, `/speckit.plan`, `/speckit.tasks`, and `/speckit.implement` to move from intent to implementation.
- **CLI-first tooling:** The `specify` CLI initializes projects, checks installed tools, and installs integration assets.
- **Multiple integration targets:** The repo supports AI coding-agent integrations and exposes slash-command style workflows; Codex CLI in skills mode uses `$speckit-*` style commands.
- **Strong anti-supply-chain warning:** The maintainers warn that similarly named PyPI packages are unaffiliated; official installs should come directly from the GitHub repo.
- **Community ecosystem:** Spec Kit includes community extensions/presets for process gates, documentation, issue sync, architecture review, CI guards, multi-agent QA, cost tracking, memory loading, and more.

## Why it matters

Spec Kit is a mainstream, GitHub-backed formalization of the same direction Self-OS has been moving toward: durable specs, explicit project constitutions, generated tasks, and constrained implementation by coding agents. It is useful as both a reference implementation and a vocabulary source for improving taskOS and Hermes Night Shift workflows.

## Self-OS implications

- **taskOS alignment:** The Spec Kit flow maps cleanly onto taskOS: idea capture → spec/PRD → plan → tasks → implementation branch → review. It may be worth comparing taskOS task folders with Spec Kit's generated `.specify/` artifacts.
- **Operating contract pattern:** `/speckit.constitution` resembles the Self-OS operating contract. The idea of a repo-local constitution could be adapted for project-specific agent behavior in `/data/taskOS/tasks/<slug>/` or implementation repos.
- **Night Shift safety:** Spec Kit's explicit planning and task generation could reduce unconstrained AFK coding risk by forcing agents to implement against concrete specs and principles.
- **Skill opportunity:** If adopted, a Hermes skill could wrap Spec Kit setup/use for this user's preferred workflow: capture idea → create taskOS entry → run/specify equivalent → generate issues → spawn implementation agents → run reviewer agents.
- **Review gate opportunity:** Community extensions such as plan-review gates, architecture guards, CI guards, cleanup/fix-findings, MAQA, and multi-model review are relevant to the user's existing multi-stage QA philosophy.

## Installation notes

The project recommends installing directly from GitHub rather than PyPI:

```bash
# Persistent installation, pinned to a release tag when possible
uv tool install specify-cli --from git+https://github.com/github/spec-kit.git@vX.Y.Z

# Latest from main, less stable
uv tool install specify-cli --from git+https://github.com/github/spec-kit.git

# Alternative with pipx
pipx install git+https://github.com/github/spec-kit.git@vX.Y.Z
```

Verification and initialization examples:

```bash
specify version
specify init <PROJECT_NAME>
specify init . --integration copilot
specify check
```

## Core workflow from README

1. **Establish project principles** with `/speckit.constitution`.
2. **Create the spec** with `/speckit.specify`, focusing on what and why rather than implementation details.
3. **Create a technical implementation plan** with `/speckit.plan`, specifying stack and architecture choices.
4. **Break down into tasks** with `/speckit.tasks`.
5. **Execute implementation** with `/speckit.implement`.
