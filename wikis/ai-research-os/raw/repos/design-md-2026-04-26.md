    19|## What is DESIGN.md?
    20|
    21|> A format specification for describing a visual identity to coding agents. DESIGN.md gives agents a persistent, structured understanding of a design system.
    22|
    23|It combines **machine-readable design tokens** (YAML front matter) with **human-readable design rationale** (Markdown prose). Tokens give agents exact values; prose tells them *why* those values exist and how to apply them.
    24|
    25|---
    26|
    27|## File Format
    28|
    29|### Structure
    30|1. **YAML front matter** — Design tokens delimited by `---` fences at the top of the file.
    31|2. **Markdown body** — Design rationale organized into `##` sections.
    32|
    33|> The tokens are the normative values. The prose provides context for how to apply them.
    34|
    35|### Example
    36|```yaml
    37|---
    38|name: Heritage
    39|colors:
    40|  primary: "#1A1C1E"
    41|  secondary: "#6C7278"
    42|  tertiary: "#B8422E"
    43|  neutral: "#F7F5F2"
    44|typography:
    45|  h1:
    46|    fontFamily: Public Sans
    47|    fontSize: 3rem
    48|  body-md:
    49|    fontFamily: Public Sans
    50|    fontSize: 1rem
    51|  label-caps:
    52|    fontFamily: Space Grotesk
    53|    fontSize: 0.75rem
    54|rounded:
    55|  sm: 4px
    56|  md: 8px
    57|spacing:
    58|  sm: 8px
    59|  md: 16px
    60|---
    61|
    62|## Overview
    63|Architectural Minimalism meets Journalistic Gravitas...
    64|
    65|## Colors
    66|- **Primary (#1A1C1E):** Deep ink for headlines...
    67|```
    68|
    69|### Token Schema
    70|```yaml
    71|version: <string>          # optional, current: "alpha"
    72|name: <string>
    73|description: <string>      # optional
    74|colors:
    75|  <token-name>: <Color>
    76|typography:
    77|  <token-name>: <Typography>
    78|rounded:
    79|  <scale-level>: <Dimension>
    80|spacing:
    81|  <scale-level>: <Dimension | number>
    82|components:
    83|  <component-name>:
    84|    <token-name>: <string | token reference>
    85|```
    86|
    87|### Token Types
    88|
    89|| Type | Format | Example |
    90||------|--------|---------|
    91|| **Color** | `#` + hex (sRGB) | `"#1A1C1E"` |
    92|| **Dimension** | number + unit (`px`, `em`, `rem`) | `48px`, `-0.02em` |
    93|| **Token Reference** | `{path.to.token}` | `{colors.primary}` |
    94|| **Typography** | object with `fontFamily`, `fontSize`, `fontWeight`, `lineHeight`, `letterSpacing`, `fontFeature`, `fontVariation` | — |
    95|
    96|### Canonical Section Order
    97|Sections are optional, but those present **must** appear in this order:
    98|
    99|| # | Section | Aliases |
   100||---|---------|---------|
   101|| 1 | Overview | Brand & Style |
   102|| 2 | Colors | |
   103|| 3 | Typography | |
   104|| 4 | Layout | Layout & Spacing |
   105|| 5 | Elevation & Depth | Elevation |
   106|| 6 | Shapes | |
   107|| 7 | Components | |
   108|| 8 | Do's and Don'ts | |
   109|
   110|### Component Tokens
   111|Components map a name to sub-token properties. Variants (hover, active, pressed) are separate entries.
   112|
   113|```yaml
   114|components:
   115|  button-primary:
   116|    backgroundColor: "{colors.tertiary}"
   117|    textColor: "{colors.on-tertiary}"
   118|    rounded: "{rounded.sm}"
   119|    padding: 12px
   120|  button-primary-hover:
   121|    backgroundColor: "{colors.tertiary-container}"
   122|```
   123|
   124|**Valid properties:** `backgroundColor`, `textColor`, `typography`, `rounded`, `padding`, `size`, `height`, `width`.
   125|
   126|### Consumer Behavior for Unknown Content
   127|
   128|| Scenario | Behavior |
   129||----------|----------|
   130|| Unknown section heading | Preserve; do not error |
   131|| Unknown color token name | Accept if value is valid |
   132|| Unknown typography token name | Accept as valid typography |
   133|| Unknown component property | Accept with warning |
   134|| **Duplicate section heading** | **Error; reject the file** |

   138|## CLI Toolkit: `@google/design.md`
   139|
   140|A single package with two entry points: a **CLI binary** and a **programmatic linter API**.  
   141|*Includes a `designmd` bin alias for Windows compatibility (`cmd.exe` confuses `.md` extensions).*
   142|
   143|### Installation
   144|```bash
   145|npm install @google/design.md
   146|# or run directly
   147|npx @google/design.md lint DESIGN.md
   148|```
   149|
   150|All commands accept a file path or `-` for stdin. Output defaults to JSON.
   151|
   152|### `lint` — Validate Structure
   153|```bash
   154|npx @google/design.md lint DESIGN.md
   155|npx @google/design.md lint --format json DESIGN.md
   156|cat DESIGN.md | npx @google/design.md lint -
   157|```
   158|
   159|| Option | Type | Default | Description |
   160||--------|------|---------|-------------|
   161|| `file` | positional | required | Path to DESIGN.md (or `-` for stdin) |
   162|| `--format` | `json` | `json` | Output format |
   163|
   164|**Exit code:** `1` if errors found, else `0`.
   165|
   166|**Example output:**
   167|```json
   168|{
   169|  "findings": [
   170|    {
   171|      "severity": "warning",
   172|      "path": "components.button-primary",
   173|      "message": "textColor (#ffffff) on backgroundColor (#1A1C1E) has contrast ratio 15.42:1 — passes WCAG AA."
   174|    }
   175|  ],
   176|  "summary": { "errors": 0, "warnings": 1, "info": 1 }
   177|}
   178|```
   179|
