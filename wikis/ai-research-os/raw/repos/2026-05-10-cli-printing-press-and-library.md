## What this is

CLI Printing Press is a generator for agent-native Go CLIs and MCP servers. The core pitch is that a good CLI is not just a developer convenience; for agents, it is a token-saving and error-reducing interface. Instead of forcing an agent to browse docs, infer API shape, and make brittle calls, the project tries to generate a focused CLI with local sync, offline search, compound commands, and companion skills.

The Printing Press Library is the catalog of generated CLIs. The extracted README summary says the library currently lists **74 CLIs across 16 categories**. Examples include ESPN, flight-goat, movie-goat, recipe-goat, Sentry, Linear, and other domain-specific CLIs.

## Core thesis

The repeated thesis across both repos:

> Nothing is more valuable than time and money. In a world of AI agents, that's speed and token spend. A well-designed CLI is muscle memory for an agent: no hunting through docs, no wrong turns, no wasted tokens.

This is directly relevant to Self-OS and Hermes. A CLI is a stable action surface for agents. It can compress a messy API into a small set of commands with predictable flags, structured output, local state, and domain-level compound queries.

## Non-obvious insight pattern

CLI Printing Press uses a "Non-Obvious Insight" framing for each generated CLI:

```text
[API] isn't just [obvious thing]. It's [non-obvious thing].
Every [data point] is a signal about [hidden truth].
```

Examples from the repo summary:

- Discord is not just chat; it is a searchable knowledge base and institutional memory.
- Linear is not just issue tracking; it is a team behavior observatory.
- Stripe is not just payments; it is a business health monitor.
- GitHub is not just code hosting; it is an engineering culture fingerprint.
- Notion is not just docs; it is a knowledge decay detector.
- Slack is not just messaging; it is an organizational nervous system.

This framing is useful because it forces a generated CLI to expose the *real operator questions*, not just CRUD endpoints.

## Why this matters for Self-OS and Hermes

This tool can help us make CLI interfaces for Self-OS/Hermes workflows.

The strongest application is not "generate a CLI for everything." The stronger move is to use the Printing Press pattern for high-value internal surfaces where agents repeatedly waste context or make mistakes:

- Self-OS wiki operations: search raw captures, list unprocessed sources, inspect wiki health, open source summaries, create capture notes.
- taskOS: create tasks, inspect task state, promote ideas into specs, summarize stale backlog.
- Hermes Kanban: create task graphs, inspect blocked tasks, summarize runs, fetch parent/child context, report workstream health.
- Cron and operating loop: inspect schedules, last status, failed jobs, next runs, delivery targets.
- Newsletter/content workflows: list week folders, candidate signals, source-map status, artifact completeness.
- Agent experiment workspaces: read `mission.md`, list artifacts, show reward-cycle state, summarize reviews.

For agents, these are better as explicit commands than as repeated ad hoc shell/file operations.

## Candidate Self-OS CLI idea

A possible first CLI target:

```text
selfos isn't just a folder of markdown. It is an operating memory and routing layer.
Every raw capture, task, cron run, Kanban card, and wiki page is a signal about what the operator is trying to build, defer, automate, or decide.
```

Possible command groups:

```text
selfos wiki unprocessed --wiki ai-research-os
selfos wiki capture-url <url> --wiki ai-research-os --type repo
selfos taskos create --from-idea <path>
selfos kanban blocked --with-context
selfos cron failures --since 7d
selfos newsletter week --current --status
selfos experiment status <slug>
selfos brief daily --kind evening
```

A narrower first target may be better:

```text
hermes-workspace isn't just an experiment folder. It is the agent's working contract.
Every mission, artifact, review, and reward-cycle note is a signal about whether the agent is progressing or merely producing output.
```

That would align with the current `/goal`, goal-buddy, `mission.md`, artifacts, and reward-cycle work.

## Installation notes from extracted README

CLI Printing Press binary:

```bash
go install github.com/mvanhorn/cli-printing-press/v4/cmd/printing-press@latest
```

Library starter pack:

```bash
npx -y @mvanhorn/printing-press install starter-pack
```

Install one tool:

```bash
npx -y @mvanhorn/printing-press install espn
```

Search/list/update examples:

```bash
npx -y @mvanhorn/printing-press search sports
npx -y @mvanhorn/printing-press list
npx -y @mvanhorn/printing-press update espn
npx -y @mvanhorn/printing-press uninstall espn --yes
```

The repo says the primary interface is the companion skill plus the binary. The binary alone can research/generate/verify/score, but the curated agent loop lives in the skills.

## Evaluation notes

Before using this for Self-OS/Hermes, evaluate:

- Does the generated CLI preserve hand edits on regeneration?
- Can output formats be made reliably machine-readable for Hermes tools?
- Can generated commands avoid over-broad API coverage and focus on high-value compound workflows?
- How much generated code needs human review?
- Can this produce a local-first CLI over markdown/git/SQLite state, not just web APIs?
- Can the generated MCP server be useful inside Hermes without bloating the tool surface?

## Suggested next step

Create a small spike: use the Printing Press pattern to design, not necessarily generate, a minimal `selfos` or `hermes-workspace` CLI. Start with 5-8 commands that remove repeated agent friction. If the command design is good, then test whether CLI Printing Press can generate or accelerate the implementation.
